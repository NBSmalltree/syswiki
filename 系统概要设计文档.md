# "系统百科"平台概要设计文档

## 1. 文档说明

### 1.1 编写目的

本文档基于《"系统百科"平台需求说明文档（PRD）》，从系统架构层面进行概要设计，明确平台的总体架构、功能模块划分、数据库设计框架、接口规范及安全策略，为后续前端/后端详细设计与开发提供全局指导。

### 1.2 读者对象

| 角色 | 关注重点 |
|------|---------|
| 项目经理 | 整体架构、里程碑、模块依赖关系 |
| 前端开发 | 应用架构、接口规范、前端模块划分 |
| 后端开发 | 技术架构、数据库设计、服务层设计、RAG方案 |
| 测试工程师 | 功能模块清单、接口清单、非功能需求指标 |
| 运维人员 | 部署架构、环境规划、安全策略 |

### 1.3 术语与缩写

| 术语 | 说明 |
|------|------|
| 系统空间 | 一个独立系统在平台中的百科实例，以 `system_id` 隔离 |
| 黄金链路 | 核心交易从发起到落地的全链路调用路径 |
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| LangChain4j | Java生态的LLM应用开发框架 |
| DeepSeek-v4 | 行内部署的大语言模型矩阵 |

## 2. 项目概述

### 2.1 项目背景

银行内网系统资产分散在不同团队的文档、Wiki、Excel中，存在信息孤岛问题。跨团队协同困难，新员工熟悉系统周期长，缺乏统一的系统知识管理平台。

### 2.2 核心目标

1. **结构化资产管理** — 统一Markdown模版，全景管理系统的简介、技术栈、服务器、网络、数据库及运维SQL
2. **可视化黄金链路** — 基于ECharts动态拓扑，展示交易的接口级、报文级全链路走向
3. **内网AI智能问答** — 基于DeepSeek-v4 + RAG技术，提供精准无幻觉的系统知识问答

### 2.3 项目范围

| 阶段 | 内容 |
|------|------|
| 首期（本期） | 以某核心系统为标杆，完成全量功能开发与数据填充 |
| 后续扩展 | 推广至全行其他系统入驻，复用同一套架构，仅新增系统空间 |

## 3. 总体架构设计

### 3.1 业务架构

```
┌─────────────────────────────────────────────────────────────┐
│                      统一门户（Web）                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 系统门户  │  │ 百科浏览  │  │ 拓扑查看  │  │ AI问答   │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
└─────────────────────────┬───────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────┐
│                     业务逻辑层                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 空间管理  │  │ 内容管理  │  │ 拓扑管理  │  │ RAG服务  │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
└─────────────────────────┬───────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────┐
│                     数据与AI层                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐      │
│  │  Oracle   │  │ 向量存储  │  │  行内 DeepSeek API   │      │
│  └──────────┘  └──────────┘  └──────────────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

**业务流程概要：**

1. 系统负责人创建"系统空间"，填写基础信息
2. 通过在线编辑或离线Markdown导入，填充百科各模块内容
3. 后台自动触发文本切片与向量化，更新RAG知识库
4. 浏览用户通过门户进入系统空间，查阅百科内容或拓扑图
5. 用户通过AI问答入口，基于系统知识库进行智能问答

### 3.2 应用架构（分层设计）

```
┌─────────────────────────────────────────────────────┐
│              前端展示层 (Vue3 + Element Plus)         │
│   Pages → Components → Store (Pinia) → API Layer    │
└───────────────────────────┬─────────────────────────┘
                            │ HTTP/REST + SSE(Stream)
┌───────────────────────────▼─────────────────────────┐
│              后端接口层 (Spring Boot)                 │
│   Controller → Service → Repository → Oracle        │
│                   │                                  │
│                   ▼                                  │
│              AI 编排层 (LangChain4j)                  │
│   TextSplitter → EmbeddingStore → DeepSeek Client   │
└─────────────────────────────────────────────────────┘
```

**前端分层：**
- **页面层（Pages）** — 对应路由的各功能页面
- **组件层（Components）** — 可复用的UI组件
- **状态层（Pinia Store）** — 全局状态管理
- **API层（Axios封装）** — 统一的后端接口调用

**后端分层：**
- **Controller层** — RESTful接口暴露，参数校验
- **Service层** — 核心业务逻辑
- **Repository层** — 数据库访问（MyBatis/JPA）
- **AI编排层** — LangChain4j封装，管理RAG工作流

## 4. 功能模块概要设计

### 4.1 系统空间管理模块

**职责：** 管理多系统入驻，每个系统拥有独立的百科空间。

**功能点：**

| 功能 | 说明 |
|------|------|
| 空间列表 | 门户页展示所有已入驻系统卡片，含名称、简称、负责人、最近更新时间 |
| 创建空间 | 填写系统基础信息（名称、简称、负责人），自动生成SYSTEM_ID |
| 进入空间 | 点击卡片进入该系统的专属百科空间，所有后续模块均在此空间上下文中 |
| 空间隔离 | 所有数据（内容、拓扑、向量）均以system_id隔离 |

**首期范围：** 仅创建一个核心系统空间，后续可扩展。

### 4.2 数据录入与更新模块

**职责：** 提供Markdown驱动的内容管理能力。

**功能点：**

| 功能 | 说明 |
|------|------|
| 模版下载 | 提供标准化Markdown模版，包含固定层级标题 |
| 在线编辑 | 集成md-editor-v3，左侧编辑 + 右侧实时预览 |
| 快速导入 | 上传.md文件，系统自动解析标题层级并填充到对应模块 |
| 一键导出 | 将当前系统全部/部分内容导出为.md文件 |
| 版本轨迹 | 每次保存自增version，记录operator和update_time |

**数据流：**
```
离线编写.md ──上传──▶ 后端解析标题 ──▶ 按MODULE_TYPE分段存储 ──▶ 触发向量化
在线编辑   ──保存──▶ 后端接收Markdown ──▶ 更新对应MODULE_TYPE ──▶ 触发向量化
```

### 4.3 核心百科内容展示模块

- 系统核心功能描述（Markdown渲染）

|--------|---------|
| 基础设施清单 | 服务器IP、内网域名、部署模块、CPU/内存 |
| 网络策略管理 | 防火墙策略状态、F5/Nginx负载均衡配置、域名映射 |

#### 4.3.4 快速接入指南

- 接入流程说明
- Maven依赖配置
- 测试环境联调注意事项

### 4.4 黄金链路与上下游拓扑模块

**职责：** 基于ECharts实现动态可交互的系统拓扑图。

**核心设计：**

| 层级 | 说明 |
|------|------|
| 全局视角 | 展示当前系统在上下游中的位置，核心交易的端到端链路 |
| 节点定义 | 系统节点（名称、类型、环境标识） |
| 连线定义 | 通信协议、接口名称、交易码 |
| 下钻交互 | 点击节点或连线，侧边栏弹出接口级/报文级详情 |

**数据结构（拓扑配置JSON）：**

```json
{
  "nodes": [
    { "id": "node1", "name": "核心路由网关", "type": "gateway", "x": 100, "y": 200 },
    { "id": "node2", "name": "跨境支付通", "type": "system", "x": 400, "y": 200 }
  ],
  "edges": [
    {
      "source": "node1",
      "target": "node2",
      "protocol": "HTTP/HTTPS",
      "interfaceName": "/api/cross-border/pay",
      "interfaceDetails": "Markdown格式的接口与报文详情"
    }
  ]
}
```

### 4.5 AI智能问答模块

**职责：** 基于RAG技术实现系统知识库的智能问答。

**架构流程：**

```
用户提问 ──▶ 意图识别/复杂度判断 ──┬──▶ 简单问题 ──▶ DeepSeek-Flash ──▶ 流式回答
                                   └──▶ 复杂问题 ──▶ DeepSeek-Think  ──▶ 流式回答
                                              │
                                              ▼
                                   向量检索(当前system_id) ──▶ 上下文注入Prompt
```

**核心策略：**

| 策略 | 说明 |
|------|------|
| 空间隔离 | 检索范围严格限定当前system_id的向量空间 |
| 无幻觉约束 | Prompt硬约束：仅基于检索到的上下文回答，未收录则提示联系负责人 |
| 动态模型调度 | 常规问答用Flash（低延迟），复杂推演用Think-High/Max（深度推理） |
| 流式响应 | 基于SSE（Server-Sent Events）实现打字机效果 |
| 知识同步 | Markdown内容更新时，自动触发文本切片→Embedding→向量存储更新 |

## 5. 数据库概要设计

### 5.1 设计原则

1. **多系统通用** — 所有表结构均包含 `SYSTEM_ID` 字段，新系统入驻无需修改表结构
2. **版本可追溯** — 内容类表预留 `VERSION`、`OPERATOR`、`UPDATE_TIME` 字段
3. **大文本分离** — Markdown源码使用CLOB类型存储，与结构化元数据分离
4. **向量存储解耦** — 向量数据存储在独立的向量存储中（非Oracle），通过system_id关联

### 5.2 ER关系概览

```
SYS_ENCY_SPACE (系统空间主表)
    │
    ├── 1:N ──▶ SYS_ENCY_CONTENT (百科内容明细)
    │              按MODULE_TYPE区分不同模块内容
    │
    ├── 1:N ──▶ SYS_ENCY_TOPOLOGY (拓扑链路配置)
    │              存储节点与连线关系
    │
    └── 1:N ──▶ SYS_ENCY_SQL_LIBRARY (运维SQL库)
                   按类别分组管理SQL/Shell指令

SYS_ENCY_CONTENT_VERSION (内容版本历史)
    │
    └── N:1 ──▶ SYS_ENCY_CONTENT (关联内容明细)
```

### 5.3 核心表清单

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| SYS_ENCY_SPACE | 系统空间主表 | SYSTEM_ID, SYSTEM_NAME, SYSTEM_CODE, OWNER |
| SYS_ENCY_CONTENT | 百科内容明细 | CONTENT_ID, SYSTEM_ID, MODULE_TYPE, MD_CONTENT, VERSION |
| SYS_ENCY_TOPOLOGY | 拓扑链路配置 | LINK_ID, SYSTEM_ID, FROM_NODE, TO_NODE, PROTOCOL |
| SYS_ENCY_SQL_LIBRARY | 运维SQL库 | SQL_ID, SYSTEM_ID, CATEGORY, SQL_TEMPLATE, PARAMS |
| SYS_ENCY_CONTENT_VERSION | 内容版本历史 | VERSION_ID, CONTENT_ID, VERSION, MD_CONTENT, OPERATOR |

### 5.4 MODULE_TYPE 枚举规划

| 枚举值 | 对应模块 | 说明 |
|--------|---------|------|
| INTRO | 系统简介与技术栈 | 系统核心功能描述 |
| ARCH_TEST | 测试环境架构 | 测试环境物理/逻辑/信创架构 |
| ARCH_PROD | 生产环境架构 | 生产环境物理/逻辑/信创架构 |
| SERVER | 服务器配置 | 服务器IP、域名、部署模块等 |
| NETWORK | 网络策略 | 防火墙、负载均衡、域名映射 |
| DATABASE | 数据库配置 | 实例状态、连接池、读写分离 |
| GUIDE | 快速接入指南 | 接入流程、Maven依赖、联调说明 |

## 6. 接口概要设计

### 6.1 接口规范

- **协议：** HTTP/HTTPS（内网）
- **风格：** RESTful
- **数据格式：** JSON
- **编码：** UTF-8
- **流式响应：** AI问答接口使用SSE（Server-Sent Events）

**统一响应格式：**

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

**错误码规范：**

| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 参数校验失败 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |
| 10001 | 系统空间不存在 |
| 10002 | Markdown解析失败 |
| 10003 | AI服务不可用 |

## 7. 安全设计

### 7.1 数据安全

| 风险点 | 应对策略 |
|--------|---------|
| 敏感信息泄露 | 严禁在百科内容中明文存储生产密码、密钥；Markdown导入时增加敏感词扫描告警 |
| 数据传输安全 | 前后端通信走HTTPS（内网证书），AI API调用在行内局域网内闭环 |
| 数据隔离 | 所有查询均强制携带system_id，避免跨系统数据泄露 |

### 7.2 AI调用安全

| 风险点 | 应对策略 |
|--------|---------|
| Prompt注入 | 系统Prompt硬约束，禁止用户输入覆盖系统指令 |
| 数据外泄 | 所有AI请求/响应均在行内局域网内，不经过公网 |
| 幻觉输出 | Prompt强制要求"仅基于知识库回答"，未收录则明确拒绝 |

### 7.3 认证与授权（首期简化）

首期暂不接入SSO，采用简单的本地用户标识（operator字段记录操作人）。后续对接行内统一认证平台时，可在此基础上扩展RBAC权限模型。

### 8.1 性能设计

| 指标 | 目标值 | 设计方案 |
|------|--------|---------|
| 拓扑图渲染 | 50个节点内 < 1秒 | ECharts使用Canvas渲染器，节点数据懒加载 |

### 8.2 可扩展性设计

| 维度 | 设计方案 |
|------|---------|
| 多系统扩展 | 所有表/接口/向量空间均以system_id隔离，新系统入驻零代码改动 |
| 模块扩展 | MODULE_TYPE枚举化，新增模块只需新增枚举值和对应前端页面 |
| 向量存储扩展 | LangChain4j抽象层隔离，首期内嵌方案，后期可无缝切换PgVector/Qdrant |
| AI模型扩展 | 模型名称配置化，新增模型只需修改配置文件 |

| 维度 | 设计方案 |
